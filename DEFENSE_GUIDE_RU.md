# Руководство для доклада на защите диплома

> Тема: **Project Management System (PMS)**  
> Стек: **React + Vite (frontend), Express + MongoDB (backend), JWT в HttpOnly cookie**.

---

## 1. Кратко: что это за система и какую проблему решает

PMS — это учебная веб-система управления проектами, которая покрывает базовые процессы командной работы:
- управление проектами,
- управление задачами,
- управление пользователями и ролями,
- глобальный поиск по данным.

Ключевая задача системы — дать единое пространство, где администратор и обычные пользователи работают с проектами и задачами по ролевой модели, а данные защищены авторизацией и логически корректны (например, нельзя удалить проект с активными задачами).

---

## 2. Архитектура проекта

## 2.1 Общая схема

Проект разделен на два независимых приложения:

1. **Frontend (`frontend/`)** — SPA на React:
   - отвечает за интерфейс, маршрутизацию, состояние авторизации,
   - отправляет HTTP-запросы в backend,
   - показывает пользователю результаты и ошибки.

2. **Backend (`backend/`)** — REST API на Express:
   - принимает запросы,
   - проверяет аутентификацию/авторизацию,
   - выполняет бизнес-логику через сервисный слой,
   - работает с MongoDB через Mongoose,
   - возвращает унифицированные JSON-ответы.

Такое разделение на клиент и API упрощает масштабирование и поддержку: UI и серверная логика развиваются независимо.

## 2.2 Слои backend

На backend использована многослойная структура:

- **Routes**: соответствие URL → контроллер.
- **Controllers**: HTTP-слой (чтение `req`, формирование `res`, делегирование в сервисы).
- **Services**: бизнес-логика, работа с моделями, проверки доменных правил.
- **Models**: схемы MongoDB (Project/User/Task).
- **Middleware**: сквозные задачи (аутентификация, валидация ObjectId, обработка ошибок).

Плюс такой архитектуры: контроллеры остаются «тонкими», а сложные правила находятся в сервисах, где их проще тестировать и объяснять.

---

## 3. Запуск приложения и жизненный цикл backend

1. Точка входа — `backend/src/server.js`.
2. При старте вызывается `initApp()`:
   - подключение к MongoDB,
   - сидирование тестового администратора (если не существует).
3. После инициализации Express-приложение начинает слушать порт.

Это делает старт предсказуемым: сначала инфраструктура (БД + базовый пользователь), затем сеть.

---

## 4. Конфигурация и окружение

Конфиг централизован в `backend/src/config/env.js`:
- `PORT`,
- `MONGO_URI`,
- `JWT_SECRET`,
- `JWT_EXPIRES_IN`,
- `CLIENT_URL`,
- `COOKIE_SECURE`.

Почему важно:
- один и тот же код может работать локально и в проде,
- секреты и параметры безопасности не «зашиваются» в бизнес-код.

## 5. Аутентификация и авторизация

## 5.1 Логин

Поток логина:
1. Frontend отправляет email/password на `POST /api/auth/login`.
2. Backend (`authService.login`) ищет пользователя и сверяет пароль через bcrypt.
3. Если проверка успешна — генерируется JWT с полями `id`, `isAdmin`, `email`.
4. JWT кладется в **HttpOnly cookie** (`token`), а frontend получает sanitized-данные пользователя.

Почему cookie + HttpOnly:
- токен недоступен из JavaScript-кода страницы,
- снижается риск кражи токена через XSS.

## 5.2 Проверка сессии

После загрузки SPA (`App.jsx`) диспатчится `meThunk()`:
- запрос `GET /api/auth/me`,
- если cookie валидна — возвращается текущий пользователь,
- если нет — состояние авторизации сбрасывается.

Это дает устойчивое восстановление сессии после перезагрузки страницы.

## 5.3 Middleware безопасности

- `requireAuth`:
  - читает cookie `token`,
  - валидирует JWT,
  - записывает payload в `req.user`.

- `requireAdmin`:
  - допускает только пользователей с `isAdmin = true`.

На уровне роутов это дает строгий контроль доступа:
- задачи/проекты/поиск — только для авторизованных,
- управление пользователями — только для администратора.

## 6. Доменные сущности и модель данных

## 6.1 User

Поля по смыслу:
- `email` (уникальный логин),
- `fullName`,
- `passwordHash` (хранится только хеш),
- `isAdmin` (роль),
- `isDeleted` (soft-delete).

## 6.2 Project

Поля:
- `name`, `description`,
- `status` (статус проекта),
- `isDeleted`.

Особенность: в списке проектов backend дополнительно рассчитывает `relatedTasksCount` через aggregation `$lookup`.

## 6.3 Task

Поля:
- `title`, `description`,
- `projectId` (связь с проектом),
- `authorId`, `assigneeId`,
- `status`, `type`,
- `parentTaskId` (подзадачи),
- `comments[]`,
- `isDeleted`.

Комментарий хранится как embedded-документ внутри задачи (быстрый доступ при чтении карточки задачи).

---

## 7. Бизнес-правила (что важно проговорить комиссии)

1. **Soft-delete** для пользователей/проектов/задач:
   - запись не удаляется физически,
   - проще аудит и потенциальное восстановление,
   - все выборки фильтруют `isDeleted: false`.

2. **Нельзя удалить проект с активными задачами**:
   - проверка в `projectService.deleteProject`,
   - если есть незакрытые задачи — возвращается ошибка `400`.

3. **Создание задачи возможно только в существующем проекте**:
   - валидация ссылочной целостности в `taskService.createTask`.

4. **Права на пользователей ограничены ролью admin**:
   - `requireAdmin` на роуте users.

5. **Единый контракт API**:
   - успех: `{ success: true, data: ... }`,
   - ошибка: `{ success: false, error: { message, status } }`.

---

## 8. API: как проходит запрос (end-to-end)

Покажите комиссии один типовой сценарий — например, «создание задачи»:

1. Пользователь заполняет форму в `TasksPage`.
2. Frontend вызывает `api.post('/tasks', form)`.
3. В `api.js` запрос отправляется с `credentials: 'include'` (cookie уходит на сервер).
4. Backend route `/api/tasks` защищен `requireAuth`.
5. Контроллер `createTask` вызывает `taskService.createTask`.
6. Сервис проверяет проект и создает задачу.
7. Backend возвращает `{ success: true, data: task }` с кодом `201`.
8. Frontend обновляет список задач.

Таким образом, в проекте четко разделены ответственность UI, transport-слоя и бизнес-правил.

---

## 9. Frontend: ключевые технические решения

1. **React Router + ProtectedRoute**:
   - закрытые страницы недоступны без сессии,
   - админские разделы (users) недоступны обычному пользователю.

2. **Redux Toolkit для auth-state**:
   - `loginThunk`, `meThunk`, `logoutThunk`,
   - единый state: `user`, `checked`, `loading`, `error`.

3. **Централизованный API-клиент** (`frontend/src/services/api.js`):
   - нет копирования fetch-логики по компонентам,
   - единая обработка ошибок,
   - единый способ работы с cookie.

4. **Компоненты страниц как бизнес-виды**:
   - `ProjectsPage`, `TasksPage`, `UsersPage`, `SearchPage`.

## 9.1 Подробное объяснение `frontend/src/features/auth/authSlice.js`

Ниже — разбор файла `authSlice.js`, который управляет всей клиентской аутентификацией.

### A) Зачем вообще нужен auth slice

`authSlice` — это единый источник истины по сессии пользователя на frontend.  
Он решает сразу несколько задач:

1. Хранит текущего пользователя (`user`).
2. Хранит флаг проверки сессии (`checked`), чтобы понять: мы уже спросили backend «кто я?» или нет.
3. Хранит статус запроса (`loading`) и сообщение об ошибке (`error`) для экрана логина.
4. Синхронизирует UI-маршрутизацию с backend-cookie-сессией.

Если бы этого слоя не было, логика входа/выхода/проверки сессии была бы размазана по компонентам и стала бы сложно поддерживаемой.

### B) Что делает `createAsyncThunk`

В файле описаны три асинхронных действия:

1. `loginThunk(credentials)`  
   Делает `POST /auth/login`, получает профиль пользователя, записывает его в state.

2. `meThunk()`  
   Делает `GET /auth/me`, проверяет, валидна ли cookie-сессия.  
   Это ключевой thunk для восстановления сессии после F5.

3. `logoutThunk()`  
   Делает `POST /auth/logout`, очищает серверную cookie-сессию и убирает пользователя из state.

`createAsyncThunk` автоматически генерирует 3 стадии для каждого запроса:
- `pending` (запрос начался),
- `fulfilled` (успех),
- `rejected` (ошибка).

Именно поэтому в `extraReducers` можно детально описать реакцию state на каждую стадию.

### C) Структура состояния (state machine)

Начальное состояние:

```js
{
  user: null,
  loading: false,
  checked: false,
  error: ''
}
```

Смысл полей:

- `user`: объект текущего пользователя или `null`.
- `loading`: индикатор «идет запрос логина».
- `checked`: критически важный флаг «мы уже проверили сессию на backend».
- `error`: сообщение для UI при неуспешном входе.

Почему нужен `checked`:  
без него `ProtectedRoute` не смог бы отличить «пользователь действительно неавторизован» от «мы еще просто не успели спросить backend».

### D) Переходы состояния по событиям

#### 1. `loginThunk.pending`
- `loading = true`
- `error = ''`

Что это дает в UI:
- кнопка входа может быть заблокирована,
- старые ошибки очищаются перед новой попыткой.

#### 2. `loginThunk.fulfilled`
- `loading = false`
- `user = action.payload`
- `checked = true`

Результат:
- пользователь считается авторизованным,
- защищенные роуты становятся доступны.

#### 3. `loginThunk.rejected`
- `loading = false`
- `error = action.error.message || 'Login failed'`

Результат:
- на форме логина показывается понятная ошибка.

#### 4. `meThunk.fulfilled`
- `user = action.payload`
- `checked = true`

Результат:
- сессия восстановлена после перезагрузки.

#### 5. `meThunk.rejected`
- `user = null`
- `checked = true`

Результат:
- frontend точно знает, что активной сессии нет, и может перенаправить на `/login`.

#### 6. `logoutThunk.fulfilled`
- `user = null`

Результат:
- интерфейс сразу переключается в неавторизованное состояние.

### E) Как `authSlice` связан с маршрутизацией

Связка такая:

1. В `App.jsx` при старте вызывается `dispatch(meThunk())`.
2. `ProtectedRoute` читает `user` и `checked`.
3. Пока `checked === false`, показывается Loader («Проверяем сессию...»).
4. Когда `checked === true`:
   - если `user === null` → редирект на `/login`,
   - если `adminOnly && !user.isAdmin` → редирект на `/projects`,
   - иначе → доступ к странице.

Именно эта схема убирает «мигание» роутов и гонки между загрузкой приложения и проверкой сессии.

### F) Почему это хорошее архитектурное решение для диплома

1. **Предсказуемое поведение** — есть явная state machine.
2. **Четкое разделение ответственности** — API, state и маршрутизация не смешаны.
3. **Масштабируемость** — легко добавить refresh-сценарий, MFA, профиль, смену email и т.д.
4. **Удобство объяснения на защите** — можно показать последовательность:
   `App init → meThunk → checked/user → ProtectedRoute`.

### G) Что можно улучшить в будущем

1. Добавить явную обработку `logoutThunk.rejected` (например, все равно очищать `user` локально).
2. Хранить коды ошибок (не только message), чтобы точнее управлять UI-сообщениями.
3. Добавить отдельный статус `authInitializing`, если потребуется более сложный bootstrap.
4. При необходимости внедрить refresh-token flow и автоматическое продление сессии.

## 10. Поиск и производительность

`GET /api/search?q=...` делает параллельный поиск (`Promise.all`) по:
- проектам,
- задачам,
- пользователям (только для admin).

Ограничение `.limit(15)` снижает объем ответа и нагрузку.

---

## 11. Надежность и обработка ошибок

- Ошибки в контроллерах передаются через `next(err)`.
- Глобальный `errorHandler` формирует единый формат ответа.
- Frontend API-wrapper интерпретирует и HTTP-ошибки, и `success: false` как исключение.

Плюс: единообразное поведение ошибок в UI и удобный дебаг.

---

## 12. Что можно назвать ограничениями текущей версии

Честно озвучить на защите:

1. Нет полноценной системы тестов (unit/integration/e2e).
2. Нет пагинации/сортировок на всех списках.
3. Нет refresh-токена и ротации сессий.
4. Глобальный поиск работает по regex и может требовать индексации/оптимизации на больших объемах.
5. Нет продвинутого аудита действий (кто/когда изменил сущность).

Это не «минусы ради минусов», а понятный roadmap развития.

## 14. Готовый «скрипт» доклада (5–7 минут)

Ниже шаблон выступления, который можно читать почти дословно:

1. **Введение (30–40 сек)**  
   «Я разработал веб-систему управления проектами PMS. Цель — реализовать безопасное и расширяемое приложение с ролевым доступом, CRUD-операциями и глобальным поиском».

2. **Архитектура (1–1.5 мин)**  
   «Система разделена на frontend и backend. Frontend — React SPA с маршрутизацией и Redux для состояния авторизации. Backend — Express API с Mongoose. На backend применена слоистая архитектура: routes → controllers → services → models».

3. **Безопасность (1 мин)**  
   «Аутентификация построена на JWT, токен хранится в HttpOnly cookie. Middleware `requireAuth` валидирует токен и добавляет пользователя в `req.user`. Для административных операций используется `requireAdmin`».

4. **Бизнес-логика (1–1.5 мин)**  
   «Во всех сущностях применяется soft-delete. Для проектов реализовано доменное правило: нельзя удалить проект, если у него есть активные задачи. Для задач проверяется ссылочная целостность — задача создается только в существующем проекте».

5. **Клиент-серверное взаимодействие (1 мин)**  
   «На frontend есть единый API-wrapper, который автоматически отправляет cookie и централизованно обрабатывает ошибки. Это уменьшает дублирование кода и делает поведение приложения предсказуемым».

6. **Итог и развитие (40–60 сек)**  
   «Результат — работающее приложение с прозрачной архитектурой. В roadmap: тестирование, расширение ролей, усиление безопасности и улучшение UX».

## 15. Частые вопросы комиссии и короткие ответы

**В: Почему выбрали JWT в cookie, а не localStorage?**  
О: HttpOnly cookie лучше защищает от XSS-кражи токена.

**В: Зачем soft-delete?**  
О: Для аудита, восстановления и безопасной истории изменений.

**В: Почему логика в сервисах, а не в контроллерах?**  
О: Контроллеры остаются простыми, а бизнес-правила централизованы и легче поддерживаются.

**В: Как обеспечивается ролевой доступ?**  
О: Через `requireAdmin` и условный рендер/маршрутизацию на frontend.

**В: Что будет при росте нагрузки?**  
О: Добавить индексы, пагинацию, кеширование, наблюдаемость и масштабирование API-инстансов.
